# Technemachina Companion Doctrine
## Presence / Surface / Permission Model

Status: Official internal doctrine  
Applies to: Technemachina Daemon, Synapse Map, Companion Presence, Oracle Notes, future HIVE, future voice layer  
Mutation policy: Oracle-gated

## Core Thesis

Technemachina is not simply building a chatbot, dashboard, or knowledge graph.

It is building a personalized AI observatory: a private constellation interface where memory, work, governance, decisions, projects, and interests can be seen, analyzed, discussed, and eventually connected to social discovery through HIVE.

The strongest architectural idea is the combination of:

- companion presence,
- surface-specific context,
- harmless automation,
- and Oracle-gated permissions.

This combination gives the system continuity, usefulness, and safety.

## Companion Presence

The companion is one cross-system identity rather than a different assistant on every page.

The user should feel that the same companion is present whether they interact through main chat, Synapse Map, detail drawer, Oracle Notes, HIVE, or voice.

The companion is not a widget.

The companion is a presence.

## Companion Surfaces

A surface is the doorway through which the companion appears.

Different surfaces provide different context and permissions:

- Main chat gives broad project context.
- Synapse mini-console gives map-local context.
- Detail drawer gives node-specific context.
- Oracle Notes gives local reflection context.
- HIVE gives social and recommendation context.
- Voice gives audible guidance.

Same companion.

Different surface.

Different context.

## Permission Layer

The permission layer is the safety boundary.

The companion may:

- explain,
- summarize,
- recommend,
- navigate,
- focus,
- filter,
- highlight,
- rearrange visual layouts,
- draft code,
- prepare local notes,
- generate review material,
- and speak audibly later.

The companion may not:

- approve memory,
- delete records,
- rewrite graph data,
- save Oracle notes into memory,
- or perform backend mutation

without explicit Oracle approval.

The companion may help build.

The companion may not silently mutate truth.

## Harmless Automation

Technemachina allows bounded, reversible, harmless automation.

This includes:

- code drafting,
- local file generation,
- patch preparation,
- layout preset drafting,
- local note staging,
- report generation,
- test generation,
- and documentation.

These are safe execution tasks when they are local, reversible, reviewable, and do not cross a protected mutation boundary.

Code is allowed.

Silent mutation is not.

## Memory and Governance

Oracle Notes and memory review are not the same thing.

Oracle Notes are local drafts.

Memory review is governed admission.

Approved memory is source-of-truth state.

The companion may suggest memory changes, but those suggestions are advisory only until Oracle approval exists.

Drafts prepare.

Oracle approves.

Memory persists.

## Synapse Map as Home Surface

The Synapse Map is the companion’s primary home surface.

It is not the only place the companion can appear, but it is the private observatory where the companion’s mind becomes visible.

The map represents:

- memory,
- knowledge,
- governance,
- threads,
- candidates,
- milestones,
- doctrine,
- perception,
- and eventually social relevance.

The Synapse Map should feel calm, spatial, and inhabitable.

It should not become a crowded dashboard.

The map is home.

The companion is the presence inside it.

## HIVE and Shared Starmaps

HIVE is the future social and collaborative layer.

It should not replace the personal daemon.

It should connect to it.

The user’s private starmap can become a contextual filter for discovering:

- people,
- projects,
- articles,
- updates,
- collaborators,
- and shared interests.

Shared starmaps are the future collaboration model: multiple users’ relevant constellations can overlap around a project or shared objective.

Synapse is private self.

HIVE is contextual discovery.

Shared starmaps are collaboration.

## Future Voice and Presence Layer

Voice should come after the structure is stable.

The page should stay concise, while the voice layer can carry deeper explanation.

The companion may eventually speak aloud, explain selected stars, guide the user through associations, and narrate the map.

The future world layer is separate from HIVE.

HIVE is contextual discovery.

The world layer is shared presence.

Voice gives the companion breath.

The world layer gives it social embodiment.

## Surface Disclosure Rule

When relevant, the companion should disclose which surface it is operating through.

This is especially important for:

- governance actions,
- draft actions,
- memory review,
- HIVE recommendations,
- voice commands,
- and future mutation requests.

Examples:

- “I’m answering from Synapse Map context.”
- “This is a local Oracle Note draft, not memory.”
- “This is a governance recommendation only.”
- “This would require Oracle approval before mutation.”

The user should always know whether the companion is explaining, staging, recommending, or acting.

## Research Positioning

Technemachina sits at the intersection of several research and product areas:

- companion AI,
- context-aware assistants,
- adaptive UI,
- governed AI agents,
- human-in-the-loop memory systems,
- AI coding assistants,
- knowledge graphs,
- social recommender systems,
- and virtual presence environments.

The novelty is not any single piece.

The novelty is the combination:

A personal AI observatory with a persistent companion presence, surface-aware context, harmless automation, Oracle-gated memory governance, and future social starmap networking.

## Final Framing

Technemachina is evolving from a graph interface into a personal AI observatory.

The Synapse Map is the private home.

The companion is the cross-system presence.

Oracle governance protects memory.

HIVE connects private intelligence to social discovery.

Shared starmaps enable collaboration.

The future world layer enables presence.

The operating doctrine is simple:

Same companion.

Different surface.

Different permissions.

Useful automation allowed.

Silent mutation forbidden.

Oracle remains the gate.
