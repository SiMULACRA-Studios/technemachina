# v0.3.0 — Synapse Perception Layer

## Status

PLANNED.

v0.3.0 begins the transition from Synapse Map as a visual renderer to Synapse Map as a read-only perception system.

## Core Distinction

Renderer = eyes.  
Analytics = perception.  
Mutation = Oracle-gated.

The daemon may observe the map.  
The daemon may analyze the map.  
The daemon may explain relationships.  
The daemon may recommend structural improvements.  
The daemon may not mutate the map without explicit Oracle approval.

## Primary Goal

The Synapse Perception Layer should allow the daemon to understand the structure, meaning, and lineage of its own map without rewriting it.

The goal is not control.  
The goal is self-understanding.

## Authority Boundary

v0.3.0 must clearly separate four modes:

### 1. Observation

The daemon reads the current Synapse Map.

Allowed:
- count nodes,
- count edges,
- inspect labels,
- inspect types,
- inspect metadata,
- inspect relation strength.

### 2. Interpretation

The daemon explains what the map appears to mean.

Allowed:
- summarize clusters,
- identify likely central nodes,
- identify likely bridge nodes,
- identify isolated nodes,
- explain visible relationships.

### 3. Recommendation

The daemon may suggest improvements without applying them.

Allowed:
- suggest clearer groupings,
- suggest layout improvements,
- suggest possible galaxy anchors,
- suggest places where metadata is incomplete.

### 4. Mutation

The daemon changes the map or underlying records.

Forbidden in v0.3.0.

Mutation requires a later Oracle-gated system with dry-run preview, Decision Ledger entry, rollback path, and explicit confirmation.

## Perception Hierarchy

The map should be analyzed across four levels:

### 1. Galaxy

A major domain of the daemon’s system.

Examples:
- Memory System
- Decision Ledger
- Thread Registry
- Synapse Map
- Provider / Brain Layer
- Knowledge Ingest
- Milestones
- Doctrine

Galaxy labels are starting anchors, not permanent ontology. The daemon may refine grouping logic over time as the graph becomes richer.

### 2. Sub-Galaxy

A nested cluster inside a galaxy.

A sub-galaxy is not a competing taxonomy. It is a smaller interpretive grouping contained within a larger galaxy.

Examples:
- Memory Candidates inside Memory System
- Memory Review Queue inside Memory System
- Approved Memory Records inside Memory System
- Provider Health inside Provider / Brain Layer
- Synapse Renderer inside Synapse Map
- Synapse Analysis inside Synapse Map
- Roadmap Milestones inside Milestones

### 3. Node

An individual signal inside the map.

Examples:
- a memory record
- a thread
- a decision
- a module
- a milestone
- a knowledge candidate
- a provider status object

### 4. Edge

A relationship between two signals.

Edges should be explainable.

The daemon should be able to answer:
- why this node is connected
- why this edge exists
- what system relationship the edge represents
- whether the edge is structural, semantic, temporal, or lineage-based

## Release Priorities

v0.3.0 should ship in this order:

1. Top-level map analysis.
2. Cluster / galaxy summaries.
3. Node and edge explanation.
4. Central node detection.
5. Bridge node and bridge edge detection.
6. Isolation detection.
7. Basic lineage / provenance tracing.

Layout suggestion mode should remain a later sub-release unless the first analysis layer is already stable.

## v0.3.0 First Implementation Target

The first implementation should focus on one umbrella endpoint:

- GET /synapse/analysis

This endpoint should return a read-only overview of the current map.

Expected first response shape:

- meta
- doctrine
- totals
- galaxy summaries
- central nodes
- bridge candidates
- isolated nodes
- confidence notes
- limitations

Specialized endpoints may come later, but the first release should prove the perception model through the umbrella response.

## Core Capabilities

### 1. Top-Level Map Summary

The daemon should summarize the current Synapse Map in plain English.

It should report:

- total nodes
- total edges
- major entity types
- major relation types
- strongest visible systems
- sparse or incomplete areas

### 2. Cluster / Galaxy Detection

Detect major semantic groupings in the Synapse Map.

The daemon should identify which nodes belong together and summarize each galaxy in plain English.

### 3. Hierarchical Cluster Summary

Summarize the map at multiple levels:

- galaxy summary
- sub-galaxy summary
- selected node summary
- selected edge explanation

This prevents the map from becoming a flat graph with no interpretive depth.

### 4. Central Node Detection

Identify the most important nodes in the map.

Central nodes may be:

- heavily connected
- referenced by many systems
- structurally important
- semantically important
- important because they connect multiple galaxies

### 5. Bridge Node Detection

Identify nodes that connect otherwise separate galaxies.

Bridge nodes may represent important cross-system concepts such as:

- Decision Ledger
- Memory Taxonomy
- Thread Registry
- Synapse Map
- Knowledge Ingest
- Provider Routing
- Project Context
- Milestone Clusters

### 6. Bridge Edge Detection

Identify the specific edges that connect galaxies.

This matters because sometimes the bridge is not only the node but the relationship itself.

Example:

Memory Taxonomy → Decision Ledger

The edge exists because memory classification decisions require review, governance, and traceable approval.

### 7. Bridge Strength Ranking

Rank bridge nodes and bridge edges by importance.

Not every bridge matters equally.

A bridge may be stronger if it:

- connects major galaxies
- carries many dependency paths
- connects governance to action
- links memory to decision
- links roadmap to implementation

### 8. Isolation Detection

Identify isolated or weakly connected nodes.

These may represent:

- orphaned memory candidates
- stale records
- incomplete milestones
- disconnected knowledge fragments
- modules that exist but are not integrated into the system story

### 9. Basic Lineage / Provenance Tracing

Frame lineage as auditable provenance.

The daemon should trace how a signal emerged from source to outcome when explicit edges support the path.

Example questions:

- What led to this milestone?
- Which decisions created this module?
- Which thread produced this memory?
- Which memory records influenced this roadmap?
- Which prior version introduced this capability?

Lineage should explain dependency, origin, and consequence.

### 10. Why-This-Is-Connected Explanations

The perception layer should support two related questions:

- Why is this node connected?
- Why does this edge exist?

This is more precise than a generic “why is this connected?” feature.

## Confidence and Provenance

Analytical outputs should distinguish between:

- Strong inference
- Moderate inference
- Tentative inference

If the map is incomplete or sparse, the daemon should say so.

Confidence should be based on:

- number of supporting edges
- relation strength
- metadata completeness
- explicit relation type
- whether the conclusion is directly represented or inferred

Perception should be explainable, not overconfident.

## Minimal API Surface

v0.3.0 should start small.

Required first endpoint:

- GET /synapse/analysis

Optional later endpoints:

- GET /synapse/analysis/clusters
- GET /synapse/analysis/bridges
- GET /synapse/analysis/lineage?node_id=
- GET /synapse/analysis/explain-edge?source=&target=

The umbrella endpoint should return the top-level overview by default. Specialized endpoints should refine, not replace, the umbrella response.

## Safety Rules

The Synapse Perception Layer is read-only by default.

Forbidden in v0.3.0:

- node mutation
- edge mutation
- node deletion
- edge deletion
- relation weight editing
- automatic reorganization
- automatic memory reshaping
- hidden structural changes

All future mutation must require:

- Oracle approval
- dry-run preview
- Decision Ledger entry
- rollback path
- explicit confirmation

## Frontend Ideas

Possible analysis UI elements:

- Analyze Map button
- Galaxy Summary panel
- Central Signal badges
- Bridge Signal list
- Isolated Signal warnings
- Confidence label on generated analysis
- Why Node Connected button
- Why Edge Exists button
- Lineage trail under selected node

Layout suggestion UI should wait until the first perception layer is stable.

## Definition of Done

v0.3.0 is complete when:

- The daemon can analyze the current Synapse Map without mutating it.
- The daemon can return a useful response from GET /synapse/analysis.
- The daemon can summarize major galaxies.
- The daemon can identify central nodes.
- The daemon can identify bridge candidates.
- The daemon can identify isolated nodes.
- The daemon can label analysis confidence.
- The daemon can explain at least one visible relationship from existing edge data.
- The frontend can display the analysis in a readable way.
- No mutation capability is introduced.

## Doctrine Lock

Synapse Perception improves understanding, not control.

The map may be seen.  
The map may be analyzed.  
The map may be interpreted.  
The map may be questioned.  
The map may be recommended upon.  
The map may not be rewritten without Oracle approval.
